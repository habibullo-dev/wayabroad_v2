# AI Web-Search Data Validation + Calculator Recalibration — Design

**Date:** 2026-05-31
**Status:** Approved (design) — pending spec review → implementation plan
**Owner context:** Verified by the owner (a Korea University student) that the live numbers are
not close to reality. The admission calculator is also "mediocre."

---

## 1. Problem

Two coupled problems, both rooted in **data**:

1. **The displayed facts are wrong.** Tuition, requirements, deadlines, and rankings were
   seeded synthetically or from a partial v1 import. The owner confirmed Korea University's
   numbers are off.

2. **The calculator only *looks* like a data model.** [lib/probability/score.ts](../../../lib/probability/score.ts)
   blends a rules prior with `admission_records` via "Bayesian shrinkage." But those records are
   synthetic, generated by [database/gen_db.py:209](../../../database/gen_db.py) with the formula
   `p = TIER_SEL[tier] + margin*0.35` then `×0.25` on language fail — **the same formula the engine
   uses as its prior** ([score.ts:169](../../../lib/probability/score.ts)). So the "empirical
   outcomes" layer and the "X of N similar applicants admitted" driver are **circular** — the engine
   reads back its own generator plus noise. The confidence band is a fixed ± by row count, not a real
   interval. Net: real-looking signal that is actually hollow.

## 2. Goals / Non-goals

**Goals**
- Validate & correct our published facts against the live web, **with a source URL + date for every
  number**, written into the existing `universities.verified` overlay.
- Pull real **acceptance / international-admission rates** where published, to give the calculator
  honest base rates.
- Recalibrate the calculator: remove the circular synthetic blend; base it on validated (cited) rates
  with a clearly-labeled fallback; present an honest estimate + band + Reach/Match/Safety + cited drivers.

**Non-goals (explicitly out of scope)**
- **Per-applicant admit/reject outcomes.** Not web-searchable; no school publishes "this GPA+IELTS got
  in." A future student-reported-outcomes capture is a separate track, not this spec.
- A trained ML model. Premature without real outcome data.
- A human review/approval UI. Per owner decision, AI output **auto-publishes as a labeled estimate**.

## 3. Locked decisions

| Decision | Choice |
|---|---|
| What the validator produces | Cited facts **and** competitiveness stats → the `verified` overlay |
| AI provider | **OpenAI** (Responses API + `web_search` tool), behind a swappable interface |
| Trust model | **Auto-publish, all as "estimate"** — AI data is never promoted to green "verified" fact |
| Run model | **Batch script**, run on demand by the owner (not per request) |
| Rollout | **Pilot** (Korea U + ~5) → verify → all 50 |
| Scope | One combined spec covering both the validator and the calculator rework |

## 4. Architecture

### 4.1 Validator module — `lib/validation/`
- `web-search.ts` — a thin, provider-swappable interface: `webSearchExtract(prompt, schema) →
  { value, source_url, source_date, confidence } | null`. OpenAI implementation calls the Responses
  API with the `web_search` tool and forces structured JSON.
- **Anti-hallucination contract:** the prompt instructs the model to return `null` when it cannot find
  a figure with a citation — never to guess. A value without a usable `source_url` is dropped.
- `prompts.ts` — one focused prompt per field group (asks for the *current* figure for *international*
  students, with the official source preferred).
- Server-only. `OPENAI_API_KEY` read via the existing server env module; `IS_OPENAI_CONFIGURED` flag.

### 4.2 Batch script — `scripts/validate-data.ts` (`pnpm validate:data [slug…]`)
- Targets all universities, or only the slugs passed (pilot).
- Per university, validates these field groups:
  - **Costs:** intl undergrad tuition, grad tuition, dorm, monthly living, application fee.
  - **Requirements:** TOPIK / IELTS / TOEFL minimums, application deadlines.
  - **Reputation:** QS world ranking.
  - **Competitiveness:** overall acceptance rate and international-admission rate (program-level if
    available, else university-level).
- Maps results into the `verified` overlay (extended shape, §4.3), tags every field
  `status: "ai_web"`, stamps a record-level `validated_on` date, and **upserts** to
  `universities.verified` via the service role. Idempotent.
- Prints a **before/after diff** (old value → new value + source) to stdout and a report file, so the
  owner can eyeball every change even though it auto-publishes.
- Logs OpenAI token usage / approximate cost; respects a small concurrency cap + retry.

**Merge rule (resolves AI-vs-existing conflicts):**
- A field currently tagged `verified_official` (human/official-verified, e.g. the 9 tuitions imported
  from `files/`) is **not overwritten** by an AI estimate — human verification outranks AI. But if the
  AI's cited finding **differs materially**, the script logs a **discrepancy** in the diff so the owner
  can investigate (the owner found some official-tagged numbers wrong, so these flags matter).
- All other fields (pending / `verified_thirdparty` / `estimated_*` / `ai_web` / absent) are refreshed
  to the new `ai_web` estimate when a cited value is found; left untouched when the AI returns null.

### 4.3 Schema / types
- Extend `VerifiedOverlay` ([lib/data/verified.ts](../../../lib/data/verified.ts)) with:
  - `acceptance?: { rate: number|null; intl_rate: number|null } & provenance`
  - `deadlines?: { text: string } & provenance` (optional, if reliably found)
  - record-level `validated_on?: string` and `validated_via?: "ai_web"`.
- `fieldTrust()` gains `"ai_web" → "estimate"`. No DB migration needed — `verified` is already a JSONB
  column; only the TS shape + loader change.
- The estimate chip copy gains an "AI-validated" affordance (e.g. tooltip/footnote: "Validated via web
  search on <date> — confirm with the university").

### 4.4 Calculator rework — `lib/probability/score.ts`
- **Remove** the `admission_records` empirical blend and the "outcomes of similar applicants" driver
  (the circular part). The `admission_records` table stays for a future real-outcomes track but is no
  longer read by the engine.
- **Base rate:** use the validated `acceptance.intl_rate` (or `rate`) as the prior when present —
  cited in a driver. When absent, fall back to a **clearly-labeled** coarse prior (kept from
  `TIER_BASE`, but the driver says "no published rate found — rough estimate").
- **Fit adjustments:** GPA margin and language **strength** (margin above the cutoff, not just
  pass/fail) move the estimate around the base rate. Eligibility hard-gates stay (language not met /
  GPA well below min).
- **Output:** `percent` (model estimate) + `band` (wider on the fallback prior) + `Reach | Match |
  Safety` label + drivers, at least one of which cites the acceptance-rate source when used. The
  `confidence` reflects *whether the base rate is validated-cited vs a fallback prior* — not synthetic
  sample size.
- `ProbabilityResult` extends with `basis: "validated_rate" | "prior"` and `category:
  "reach"|"match"|"safety"`; existing consumers (free check, shortlist) updated.

## 5. Data flow

```
pnpm validate:data korea-university
  → for each field group: webSearchExtract(prompt) ──(OpenAI + web_search)──▶ {value, source_url, date}
  → map → verified overlay (status: "ai_web", validated_on)
  → upsert universities.verified (service role)  +  print diff report
Live app:
  university detail → getVerified() → amber "Estimate · source" (existing gating)
  calculator → getVerified().acceptance → base rate (cited)  |  else labeled prior
```

## 6. Honesty & correctness guarantees
- Every AI number carries a real `source_url` or it is dropped (no uncited values).
- AI data renders **only** as a labeled estimate — never the green "verified" fact treatment, which
  stays reserved for human/official verification.
- The calculator never again presents fabricated "empirical outcomes."
- The diff report makes every automated change auditable by the owner.

## 7. Cost & operations
- ~7 field groups × 50 universities ≈ a few hundred web-search calls — run on demand by the owner,
  not in the request path. Token/cost logged. Pilot first to tune prompts before the full run.

## 8. Testing
- Validator mapping: `null`/uncited → dropped; `ai_web` → "estimate"; provenance preserved.
- `web-search.ts`: OpenAI client mocked (no live calls in tests); schema-validation + null path.
- Reworked `scoreAdmission`: monotonic in GPA/language; uses validated base rate when present; labeled
  fallback when absent; eligibility gates intact; no `admission_records` dependency. Existing 9 engine
  tests updated.

## 9. Rollout
1. Build validator + script; run on **Korea University** only; owner verifies accuracy.
2. Tune prompts; run the ~5-university pilot.
3. Run all 50; review the diff; ship.
4. Calculator rework lands alongside (consumes validated rates immediately for piloted unis, labeled
   fallback elsewhere).

## 10. Risks
- **Web search may not find program-level intl acceptance rates** for many Korean universities → the
  calculator's labeled-prior fallback will be common. Acceptable and honest; still better than circular
  synthetic data.
- **AI mis-citation.** Mitigated by the null-rather-than-guess contract, source-required rule,
  estimate-only labeling, and the owner-auditable diff.
- **OpenAI as a second provider** adds a key + SDK; isolated behind the validator interface.
